# 行业工作内容差异与个人学习路线

本文整理关于模块化架构、E2E 架构下行业工作内容差异，以及面向数据闭环 / 智驾应用软件开发工程师的学习路线建议。

---

## 1. 对行业工作内容的总体理解

本次讨论形成的判断是：

```text
模块化架构更偏软件工程、规则系统、实时控制和工程稳定性；
E2E 架构更偏数据工程、模型训练、闭环评测和大规模平台能力；
但两者都离不开数据，也都离不开工程。
```

更准确地说：

```text
模块化架构：工程规则和模块接口更显性，数据服务于各个子模块优化。
E2E 架构：数据和模型训练成为主战场，但工程底座、部署优化、安全验证仍然非常重。
```

---

## 2. 模块化架构下的工作内容

传统模块化自动驾驶更像一个大型实时软件系统。

典型工作包括：

```text
传感器 Driver
中间件通信
Topic / Channel 管理
时间同步
坐标转换
感知模块
预测模块
规划模块
控制模块
规则系统
状态机
仿真回放
日志录制
系统监控
故障降级
```

更偏向：

```text
模块搭建
接口设计
规则约束
实时系统
C++ 工程
中间件
性能优化
系统稳定性
```

工程师可能做：

```text
规划状态机
变道决策规则
AEB 触发逻辑
传感器融合模块
控制器调参
Cyber RT / ROS2 通信
日志解析和回放工具
```

这种工作更接近：

```text
大型复杂系统工程 + 自动驾驶业务规则 + 实时软件开发。
```

---

## 3. 模块化架构也需要数据

需要修正一个误解：

```text
模块化架构不是不需要大量数据，而是数据更多按模块、按任务使用。
```

模块化系统中很多模块也是模型驱动：

```text
感知模型需要训练数据
车道线模型需要训练数据
红绿灯模型需要训练数据
预测模型需要轨迹数据
Occupancy 模型需要多传感器数据
学习型规划也需要专家轨迹
```

数据使用方式更分散：

```text
感知漏检 → 补感知数据
车道线错误 → 补车道线标注
预测错误 → 补轨迹交互样本
规划失败 → 补规则或补场景库
```

因此：

```text
模块化架构的数据闭环是“局部修模块”；
E2E 架构的数据闭环是“整体修驾驶行为”。
```

---

## 4. E2E 架构下的工作内容

E2E 的核心目标是训练一个更统一的“驾驶大脑”。

它关心：

```text
多源传感器输入
历史帧
导航目标
地图语义
自车状态
交通参与者交互
驾驶轨迹
风险判断
```

让模型学习：

```text
当前场景是什么？
未来可能发生什么？
自车应该怎么开？
```

因此 E2E 方向工作更偏向：

```text
数据采集
数据清洗
数据标注
场景挖掘
相似场景召回
训练集构建
模型训练
模型评测
仿真回归
模型部署
线上回流
```

核心循环：

```text
发现 badcase
  ↓
召回相似场景
  ↓
补充标注 / 自动标注
  ↓
构建训练集和回归集
  ↓
训练大模型
  ↓
离线评测 / 仿真评测 / 影子模式
  ↓
上线灰度
  ↓
继续回流
```

---

## 5. E2E 也高度工程化

E2E 不是训练完一个模型就结束。

它仍然需要大量工程能力：

```text
车端部署
模型推理加速
TensorRT / ONNX 优化
NPU / GPU 调度
实时推理
多传感器同步
数据管线
训练平台
评测平台
仿真平台
模型版本管理
安全监控
轨迹检查
AEB 兜底
灰度发布
线上监控
```

量产上车要求：

```text
低延迟
高稳定性
可回滚
可监控
可解释到一定程度
安全可验证
资源占用可控
```

所以 E2E 的工程难度不是消失，而是从“写很多人工规则”转移到：

```text
大规模数据工程
大模型训练工程
车端模型部署工程
安全验证工程
```

## 6. 两种架构下岗位差异

| 维度 | 模块化架构 | E2E 架构 |
|---|---|---|
| 核心对象 | 模块、接口、规则、状态机 | 数据、模型、训练、评测 |
| 主要逻辑 | 人工拆解任务并组合模块 | 模型学习整体驾驶策略 |
| 数据使用 | 分模块数据集 | 大规模场景级数据集 |
| 工程重点 | 实时系统、C++、中间件、规划控制 | 数据管线、训练平台、模型部署 |
| 典型问题 | 某个模块错了 | 整体驾驶行为错了 |
| 迭代方式 | 修模块、调规则、补局部数据 | 挖场景、补数据、重训模型、回归验证 |
| 人员结构 | 软件工程、规划控制、系统工程 | 数据工程、算法训练、MLOps、评测仿真 |
| 仍然需要 | 数据、模型训练 | 实时工程、安全兜底、部署优化 |

## 7. 数据闭环工程师的价值位置

数据闭环 / 智驾应用软件开发工程师位于两种架构的交界处。

它不直接等同于算法研究，也不只是普通后端开发。

核心价值是：

```text
让车端产生的数据，变成可检索、可标注、可训练、可评测、可回归的数据资产。
```

典型工作包括：

```text
车端日志解析
rosbag / record / MCAP 处理
多源数据清洗
时间戳和 topic 质量检查
数据接入服务
对象存储和元数据管理
场景挖掘规则
相似场景检索
数据集版本管理
评测结果聚合
仿真回归数据构建
模型版本和数据版本绑定
```

这些能力无论模块化还是 E2E 都重要。

## 9. 分阶段学习建议

### 9.1 第一阶段：车端数据链路基础

目标：看懂智能车数据如何流动。

学习：

```text
ROS2 / DDS
Apollo Cyber RT
Topic / Channel
Publisher / Subscriber
rosbag / record / MCAP
Protobuf
Camera / LiDAR / Radar / IMU / CAN 基础数据结构
时间戳
坐标系 frame_id
```

练习项目：

```text
扫描日志包中的 topic 列表
统计每个 topic 的频率、消息数量、起止时间
按时间窗口裁剪日志
按 topic 白名单过滤日志
```

### 9.2 第二阶段：数据清洗与质量分析

目标：判断数据是否可信、适合做什么。

学习：

```text
文件完整性检查
topic 完整性检查
时间戳单调性检查
丢帧率统计
传感器质量检查
标定版本匹配
元数据清洗
数据分级
```

练习项目：

```text
日志质量评分工具
多传感器时间同步检查工具
Camera 图像质量检测工具
topic 缺失和丢帧分析报表
```

### 9.3 第三阶段：场景挖掘

目标：从海量日志中找到有价值数据。

学习：

```text
规则引擎
时序窗口分析
TTC / jerk / 急刹 / 接管挖掘
地图位置检索
ClickHouse 查询
Elasticsearch 检索
Embedding 相似检索
数据去重
主动学习
```

练习项目：

```text
急刹场景挖掘工具
接管前后上下文提取工具
badcase 相似场景召回 Demo
场景标签统计看板
```

### 9.4 第四阶段：数据集与评测闭环

目标：把数据转成训练、评测、回归资产。

学习：

```text
COCO / KITTI / nuScenes / OpenLABEL
训练集 / 验证集 / 测试集 / 回归集构建
数据集版本管理
模型版本绑定
评测指标管理
MLflow / DVC / LakeFS
```

练习项目：

```text
标注格式转换工具
数据集版本 manifest 生成器
评测结果聚合报表
模型-数据-指标追溯 Demo
```

### 9.5 第五阶段：仿真回灌

目标：把真实问题变成可复现仿真场景。

学习：

```text
OpenDRIVE
OpenSCENARIO
OSI
轨迹插值
坐标转换
场景泛化
仿真回归指标
```

练习项目：

```text
从接管片段提取场景
生成 OpenSCENARIO
绑定 OpenDRIVE 地图
做速度、光照、参与者行为扰动
```

## 11. 对职业方向的判断

如果偏模块化工程：

```text
更适合深入 C++、中间件、实时系统、规划控制、传感器融合、系统稳定性。
```

如果偏 E2E 数据闭环：

```text
更适合深入数据工程、场景挖掘、训练平台、评测平台、仿真回归、MLOps。
```

如果目标是长期竞争力：

```text
不要只押注某个单点技术，而要掌握数据链路 + 工程系统 + 模型闭环之间的连接能力。
```

## 12. 最终结论

你的理解可以总结并修正为：

```text
模块化更偏“软件工程 + 规则系统 + 实时控制”；
E2E 更偏“数据工程 + 模型训练 + 闭环评测”；
但两者都离不开数据，也都离不开工程。
```

对数据闭环 / 智驾应用软件开发工程师来说，最有价值的能力是：

```text
理解车端数据链路；
能处理多源日志；
能做数据清洗和质量评估；
能做场景挖掘和样本筛选；
能把数据组织成训练集、评测集、回归集；
能支撑模型训练、仿真验证和线上回流。
```

最终经验：

```text
未来行业越走向 E2E，越需要强大的数据闭环；
越追求量产安全，越离不开工程化实时系统。
真正稀缺的人才，是能同时理解数据、模型、工程和业务场景的人。
```
