# 智能驾驶架构演进：模块化与 E2E

本文整理关于传统工程化模块化架构与 E2E 架构的讨论，并总结两种架构在行业中的定位、差异和演进方向。

---

## 1. 传统工程化模块化架构

传统智能驾驶系统可以理解为一个多模块协同的大型实时系统。

典型链路是：

```text
传感器
  ↓
Driver / 数据采集
  ↓
中间件 Topic / Channel
  ↓
感知 Perception
  ↓
融合 Fusion
  ↓
预测 Prediction
  ↓
规划 Planning
  ↓
控制 Control
  ↓
车辆执行
```

它的核心思想是：

```text
把自动驾驶拆成多个明确子问题，每个模块负责一个相对清晰的任务。
```

例如：

```text
感知模块：识别车辆、行人、车道线、红绿灯、可行驶区域
融合模块：把相机、激光雷达、毫米波雷达结果统一成障碍物列表
预测模块：判断其他交通参与者未来可能怎么走
规划模块：决定自车未来轨迹
控制模块：把轨迹转成方向盘、油门、刹车控制量
```

---

## 2. 模块化架构的工程优势

模块化架构的优势是非常工程化的。

### 2.1 可解释

每个模块都有可观察输出：

```text
目标框
车道线
红绿灯状态
障碍物速度
预测轨迹
规划轨迹
控制命令
```

当车辆出现问题时，可以逐层排查：

```text
相机有没有丢帧？
感知有没有漏检？
融合有没有错配？
预测有没有误判？
规划是否过于激进或保守？
控制是否跟踪失败？
```

### 2.2 可调试

模块化系统可以单独回放、单独测试、单独替换。

例如：

```text
固定感知输出，测试规划模块
固定规划轨迹，测试控制模块
回放同一段 rosbag / record，比较不同模型版本结果
```

### 2.3 可验证

每个模块有明确指标：

```text
感知：Precision / Recall / mAP / 漏检率 / 误检率
预测：ADE / FDE / 轨迹命中率
规划：碰撞率 / 压线率 / 舒适性 / 通过率
控制：横向误差 / 纵向误差 / jerk
系统：延迟 / 掉帧 / CPU / GPU / 内存
```

### 2.4 适合工程分工

传统组织结构往往天然对应模块边界：

```text
感知团队
定位团队
预测团队
规划团队
控制团队
仿真团队
数据平台团队
中间件团队
```

## 3. 模块化架构的局限

模块化不是没有问题，它的核心问题在于：

```text
人类定义的模块边界，会造成信息损失和复杂规则堆叠。
```

### 3.1 信息瓶颈

原始图像、点云中有大量上下文信息，但经过模块抽象后，下游可能只看到：

```text
目标框
类别
速度
轨迹
置信度
```

原始场景里的细微语义可能已经丢失。

### 3.2 误差传播

典型链路：

```text
感知漏检
  ↓
预测没有该目标
  ↓
规划认为前方安全
  ↓
控制继续执行
  ↓
风险事件
```

上游错误会显式传递给下游。

### 3.3 规则膨胀

复杂场景很难靠有限规则覆盖。

例如：

```text
雨夜 + 逆光 + 无保护左转 + 对向电动车横穿
施工区 + 锥桶缺失 + 临停大车遮挡 + 行人突然出现
隧道出口 + 光照突变 + 车道线消失 + 前车急刹
```

规则越写越多，就会产生：

```text
规则冲突
维护困难
局部修复导致其他场景退化
状态机复杂度失控
```

## 4. E2E 架构想改变什么

E2E 架构希望减少人工拆分和人工规则，让模型从数据中学习更统一的驾驶能力。

传统模块化是：

```text
传感器输入
  ↓
人工定义的中间任务
  ↓
人工设计的模块接口
  ↓
规划控制
```

E2E 更像：

```text
传感器输入 + 导航目标 + 历史状态
  ↓
神经网络 / 世界模型 / 端到端规划模型
  ↓
未来轨迹 / 驾驶动作
```

核心变化是：

```text
从“工程师写规则和模块接口”变成“模型从数据中学习场景理解和驾驶策略”。
```

---

## 5. E2E 的几种形态

E2E 不是单一形态，可以分成多个层级。

### 5.1 传感器到控制

最激进形式：

```text
Camera / LiDAR / Radar / Navigation
  ↓
Neural Network
  ↓
steering / throttle / brake
```

这种形式概念简单，但量产难度很高：

```text
可解释性弱
安全验证难
控制稳定性难保证
训练数据要求极高
事故复盘困难
```

### 5.2 传感器到轨迹

更现实的形式是：

```text
Camera / LiDAR / Radar / Map / Navigation / History
  ↓
Neural Network
  ↓
ego future trajectory
  ↓
Controller
  ↓
steering / throttle / brake
```

模型输出未来几秒轨迹，控制器高频跟踪。

优点是：

```text
轨迹可以被检查
轨迹可以被仿真
轨迹可以被安全模块拦截
底层控制稳定性仍由传统控制器保证
```

### 5.3 感知预测规划一体化

常见形态：

```text
多传感器输入
  ↓
统一 BEV / Occupancy / World Model
  ↓
同时输出目标、占用、预测、规划轨迹
```

它不是完全黑盒，而是把感知、预测、规划逐渐融合。

这种形态通常更接近现实产业路线。

## 6. E2E 如何处理多源数据

E2E 并不是把多源数据直接混在一起。

更常见的是：

```text
Camera Encoder：提取图像特征
LiDAR Encoder：提取点云特征
Radar Encoder：提取速度和目标特征
Map Encoder：提取道路结构
Ego State Encoder：提取自车速度、加速度、方向盘角
Temporal Encoder：融合历史帧
Planning Head：输出未来轨迹
```

流程可以概括为：

```text
不同传感器先独立编码
  ↓
统一到时空表征
  ↓
模型内部融合
  ↓
输出驾驶轨迹或策略
```

与传统架构不同的是：

```text
传统融合输出显式对象列表；
E2E 融合更多发生在模型特征空间中。
```

## 7. E2E 仍然需要工程底座

E2E 不是不要工程。

它仍然需要：

```text
传感器 Driver
中间件通信
时间同步
坐标变换
日志录制
数据清洗
车端部署
模型推理加速
控制器
安全监控
AEB 兜底
灰度发布
线上监控
```

E2E 替代或融合的是：

```text
感知、预测、规划中大量人工定义的中间模块和规则逻辑。
```

但以下部分仍必须工程化存在：

```text
实时系统
底层控制
故障诊断
安全降级
数据闭环
仿真回归
模型版本管理
```

---

## 8. 两种架构对比

| 维度 | 模块化架构 | E2E 架构 |
|---|---|---|
| 核心思想 | 人工拆解任务 | 数据学习整体驾驶策略 |
| 信息流 | 感知 → 预测 → 规划 → 控制 | 输入 → 模型 → 轨迹 / 动作 |
| 中间结果 | 清晰、可解释 | 隐式或半显式 |
| 优势 | 可调试、可验证、工程确定性强 | 表达能力强、减少信息瓶颈 |
| 难点 | 规则膨胀、误差传播 | 数据需求大、验证困难 |
| 数据使用 | 分模块优化 | 场景级、行为级整体优化 |
| 实时性 | 模块 deadline + 流水线 | 固定推理周期 + 控制器跟踪 |
| 安全 | 规则兜底、AEB、降级 | Safety Checker、AEB、最小风险策略 |

## 9. 行业经验判断

更可能成功的不是纯模块化，也不是纯 E2E，而是：

```text
E2E 能力主导、模块化工程兜底的混合架构。
```

短中期：

```text
模块化工程体系仍然是量产基础；
学习化规划、BEV、Occupancy、世界模型逐渐增强。
```

长期：

```text
更多驾驶策略会由数据驱动模型学习；
工程系统负责实时、安全、部署、验证、闭环。
```

最终竞争点不是某个架构名词，而是：

```text
数据闭环效率
长尾场景覆盖
模型泛化能力
仿真验证能力
车端部署能力
安全兜底能力
量产成本控制
```

---

## 10. 结论

```text
模块化架构用工程系统和人工规则组织驾驶能力，数据用于优化各个子模块。
E2E 架构用大规模数据和模型训练形成驾驶能力，工程系统用于支撑训练、部署、验证和安全兜底。
```

最重要的理解是：

```text
E2E 不是抛弃工程，而是把“如何理解场景、如何生成驾驶策略”更多交给模型学习；但实时控制、安全验证、数据链路和量产工程仍然不可替代。
```
