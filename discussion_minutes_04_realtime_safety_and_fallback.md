# 实时性、安全兜底与突发情况处理

本文整理关于智能驾驶系统从多源数据输入到车辆运动控制过程中如何保证实时性，以及如何应对现实突发意外情况的讨论。

核心结论：

```text
自动驾驶实时性不是让所有模块瞬间算完，而是在严格时延预算内，让感知、预测、规划、控制形成稳定闭环，并对突发风险保留快速反应通道。
```

---

## 1. 多源处理为什么一定有延迟

从传感器到车辆控制，典型链路是：

```text
传感器采集
  ↓
Driver 解析
  ↓
时间同步 / 坐标变换
  ↓
感知识别
  ↓
多传感器融合
  ↓
预测
  ↓
规划
  ↓
控制
  ↓
执行器响应
```

每一步都需要时间：

```text
Camera 曝光和传输需要时间
LiDAR 扫描一圈需要时间
Radar 目标处理需要时间
深度学习感知推理需要时间
融合和跟踪需要时间
预测和规划需要时间
控制命令下发到底盘也需要时间
```

因此真实系统不是“看到之后立即控制”，而是：

```text
基于最近可用信息 + 当前状态预测补偿 + 安全策略，做实时滚动决策。
```

---

## 2. 实时性的第一层：时延预算与 deadline

每个模块都有最大耗时要求。

典型时延可能类似：

```text
Camera 采集与传输：10-30ms
LiDAR 点云形成：50-100ms
Radar 处理：20-50ms
感知推理：20-80ms
融合跟踪：5-20ms
预测：10-30ms
规划：10-50ms
控制：5-10ms
底盘响应：几十到上百 ms
```

不同公司和硬件平台差异很大，但思想一致：

```text
总链路延迟必须可控；
单模块不能无限等待；
超过 deadline 就要用降级结果或上一帧结果。
```

工程经验：

```text
规划模块不能等所有相机、LiDAR、Radar 完全到齐再算；
它必须在当前周期 deadline 前，使用已经到达且质量合格的数据做决策；
迟到的数据进入下一周期。
```

---

## 3. 实时性的第二层：异步流水线

自动驾驶系统不是完全串行运行。

错误理解：

```text
等 Camera 完全处理完
再等 LiDAR 完全处理完
再等 Radar 完全处理完
再做融合
再预测
再规划
再控制
```

真实系统更像流水线：

```text
第 t 帧图像在做感知
第 t-1 帧感知结果在做融合
第 t-2 帧融合结果在做预测
第 t-3 帧预测结果在做规划
控制模块持续高频跟踪最新轨迹
```

特点：

```text
各模块并行运行；
每个模块用最近可用输入；
系统通过时间戳知道每个结果对应哪个时刻；
下游通过状态外推做时间补偿。
```

这牺牲一点绝对同步，换取实时性。

---

## 4. 实时性的第三层：高频控制闭环

控制模块频率通常高于感知和规划。

典型频率：

```text
感知：10 / 20 / 30Hz
规划：10 / 20Hz
控制：50 / 100Hz
IMU：100 / 200 / 1000Hz
Chassis：50 / 100Hz
```

控制模块不会等感知每次更新才执行。

典型工作方式：

```text
规划模块每 100ms 给一条未来轨迹；
控制模块每 10ms 根据当前车速、方向盘角、车辆状态跟踪轨迹；
如果下一条轨迹还没来，短时间继续使用当前轨迹或进入安全策略。
```

这保证车辆动作是连续的，而不是“感知来一帧，车才动一下”。

---

## 5. 实时性的第四层：时间戳与状态外推

假设感知结果时间戳是：

```text
10:00:00.100
```

规划拿到它时已经是：

```text
10:00:00.180
```

不能直接把目标当作仍在 100ms 前的位置。

系统需要做状态外推：

```text
目标当前估计位置 = 历史位置 + 速度 × 时间差
自车当前状态 = 历史状态 + IMU / 轮速积分
```

因此目标状态通常包含：

```text
position
velocity
acceleration
timestamp
confidence
track_id
```

核心经验：

```text
数据可以有延迟，但必须知道它是什么时刻的数据。
```

---

## 6. 实时性的第五层：任务优先级调度

系统会区分任务优先级。

高优先级：

```text
控制命令
底盘状态
IMU
AEB / 紧急制动
关键障碍物检测
实时规划
```

低优先级：

```text
日志上传
可视化
非关键诊断
离线统计
地图缓存更新
大文件写盘
```

工程上会使用：

```text
实时线程
线程优先级
CPU 绑核
GPU 任务调度
共享内存 / 零拷贝
进程隔离
QoS 队列
看门狗 watchdog
```

原则：

```text
控制和安全链路优先；
日志和可视化可以丢帧或延迟；
非关键任务不能拖垮主控链路。
```

---

## 7. 实时性的第六层：QoS 与过期数据处理

实时系统最怕旧数据堆积。

例如：

```text
500ms 前的感知结果，对高速行驶已经可能很危险。
```

每类数据都有有效期：

```text
Camera 感知结果超过阈值 → 不再用于规划
Radar 目标太久没更新 → track 降置信度或删除
定位时间戳过旧 → 进入定位降级
规划轨迹超时 → 控制模块进入安全停车或减速
```

队列策略：

```text
高频图像：丢旧帧，保留最新帧
控制指令：不能堆积旧命令
日志写盘：可以缓冲
诊断信息：可以低频处理
```

核心原则：

```text
实时控制宁可用最新的不完整数据，也不能用完整但过期的数据。
```

## 8. 突发风险：轻重路径分离

普通路径可能是：

```text
Camera / LiDAR / Radar
  ↓
感知融合
  ↓
预测
  ↓
规划
  ↓
控制
```

但突发危险不能完全依赖完整链路。

因此系统通常有快速安全通道：

```text
Radar / LiDAR / Camera 快速障碍物检测
  ↓
TTC / 碰撞风险判断
  ↓
AEB / 紧急制动 / 最小风险策略
```

例如前车突然急刹：

```text
Radar 发现相对速度快速变大
LiDAR 发现前方距离快速缩短
Camera 可能还在跑复杂识别
```

系统可以先触发：

```text
减速
预制动
AEB
规划轨迹收缩
控制进入安全模式
```

行业经验：

```text
正常驾驶靠完整感知-预测-规划链路；
紧急风险靠更短、更保守、更高优先级的安全链路。
```

---

## 9. 降级策略

传感器或模块异常时，系统不能直接崩溃，而要降级。

示例：

```text
前视相机掉线 → 降低功能等级，限制速度，依赖 Radar / LiDAR 做基础避障
LiDAR 异常 → 降低融合权重，增加相机 / Radar 权重
定位异常 → 退出高阶功能或进入最小风险停车
规划超时 → 使用上一条短期安全轨迹，并触发减速
控制异常 → 交还驾驶员或执行安全停车
```

核心原则：

```text
能力可以下降，但安全边界不能丢。
```

---

## 10. 系统监控与看门狗

实时系统必须持续监控自己。

监控指标包括：

```text
每个模块耗时
端到端延迟
topic 频率
topic 延迟
丢帧率
CPU / GPU 使用率
内存
队列积压
线程卡死
规划超时次数
控制命令超时
```

异常处理：

```text
报警
降级
重启非关键模块
切换备份
请求接管
安全停车
记录日志回传
```

系统监控是实时性的最后一道工程保障。

## 11. 能否覆盖所有突然意外情况

严格说：

```text
不能保证覆盖现实世界所有突发场景。
```

现实世界可能出现无限组合：

```text
遮挡后儿童冲出
高速异物掉落
前车急刹并变道
施工人员突然进入车道
非机动车逆行穿插
动物冲出
强光导致相机瞬时失效
雨雪导致传感器质量下降
```

系统能做的是通过三类机制尽量提高覆盖能力。

### 11.1 感知冗余

```text
Camera：语义强，能识别颜色、车道线、红绿灯、类别
LiDAR：距离和几何强，能感知 3D 结构
Radar：速度强，抗雨雾和强光
IMU / Chassis：高频自车运动状态
```

某个传感器变差时，其他传感器提供补充。

### 11.2 安全策略保守化

不确定性升高时，系统更保守：

```text
遮挡区域多 → 降速
传感器置信度下降 → 增大安全距离
路口交互复杂 → 延迟起步或缓慢探出
检测到目标不确定 → 先按潜在风险处理
```

突发场景优先级：

```text
安全性 > 舒适性 > 效率
```

### 11.3 最小风险策略

系统判断自己无法可靠处理时，应进入最小风险状态：

```text
减速
靠边
停车
打开双闪
请求驾驶员接管
退出高阶功能
```

经验判断：

```text
自动驾驶系统不一定能处理所有场景，但必须知道什么时候自己不可靠。
```

## 12. E2E 架构下的实时性

E2E 有潜在实时性优势：

```text
减少模块间串联延迟
减少人工中间表示转换
可以直接输出轨迹
```

但 E2E 不天然实时，仍受限于：

```text
传感器采集延迟
模型推理延迟
硬件算力
多帧时序输入
安全验证
控制执行延迟
```

现实 E2E 通常采用：

```text
E2E Planner：固定频率输出未来轨迹
Controller：高频跟踪轨迹
Safety Checker：实时检查轨迹风险
AEB / 最小风险策略：紧急兜底
```

E2E 主模型负责常规驾驶策略，安全监控和紧急制动通常仍保留独立快速通道。

## 13. 前方突然行人案例

车辆 50km/h 行驶，遮挡后突然有行人出现。

可能响应流程：

```text
1. Camera 新一帧捕捉到行人，但深度学习推理需要几十 ms
2. LiDAR 同时看到前方出现新的点云 cluster
3. Radar 可能对行人不稳定，但可能有弱目标
4. 融合模块发现前方近距离新目标，置信度快速上升
5. 风险模块计算 TTC 过低
6. 规划模块立即生成制动轨迹
7. 控制模块执行制动
8. 如果风险极高，AEB 快速通道直接触发紧急制动
```

关键不是完整理解：

```text
这是成年人还是儿童？
是在跑还是走？
是否一定会进入车道？
```

而是快速判断：

```text
前方存在可能碰撞的动态障碍物，必须立即减速。
```

---

## 14. 结论

从数据输入到运动控制的实时性主要靠：

```text
1. 模块时延预算和 deadline
2. 异步流水线并行处理
3. 高频控制闭环
4. 时间戳与状态外推补偿延迟
5. 安全关键任务优先调度
6. QoS 和缓存策略丢弃过期数据
7. 轻重路径分离，突发风险走快速安全通道
8. 传感器和功能降级策略
9. 系统监控、看门狗和异常处理
```

最终经验：

```text
自动驾驶不是等所有多源数据都完整处理完再控制，而是在严格时间预算内使用最新可信数据持续滚动决策，并为突发危险保留更快、更保守的安全响应通道。
```
