# 智能驾驶数据闭环

数据闭环可以概括为：

```text
把真实道路和仿真中发现的问题，转化为可检索、可标注、可训练、可评测、可回归的数据资产，并持续推动模型和系统迭代。
```

完整链路：

```text
采集 → 清洗 → 挖掘 → 标注 → 训练 → 评测 → 仿真 → 部署 → 再采集
```

---

## 1. 为什么智能驾驶需要数据闭环

智能驾驶不是一次性开发完成的软件，而是一个持续进化系统。

### 1.1 道路场景复杂

真实道路存在大量长尾场景：

```text
鬼探头
加塞
无保护左转
雨夜逆光
施工区域
异形路口
隧道出入口
环岛
临停车辆
非机动车横穿
规划犹豫
频繁接管
```

这些很难靠人工规则一次性写全。

### 1.2 模型效果高度依赖数据

无论模块化还是 E2E，都需要持续补充数据：

```text
感知模型需要更多漏检 / 误检样本
预测模型需要更多交互场景
规划模型需要更多复杂驾驶行为数据
E2E 模型需要更多完整驾驶片段和长尾场景
```

### 1.3 量产系统必须持续验证

每次升级都要确认：

```text
新模型有没有修复旧问题？
新模型有没有引入新问题？
接管率有没有下降？
误刹、漏刹、规划犹豫有没有改善？
仿真和实车表现是否一致？
```

所以数据闭环不是“数据部门的辅助工作”，而是智能驾驶系统持续迭代的核心发动机。

## 2. 数据闭环完整链路

完整数据闭环可以拆成 10 个环节：

```text
1. 车端数据采集
2. 数据上传与接入
3. 数据存储与索引
4. 数据清洗与治理
5. 场景挖掘
6. 标注与质检
7. 数据集构建
8. 模型训练
9. 评测与仿真回归
10. 部署与线上回流
```

流程图：

```text
车端运行
  ↓
日志采集
  ↓
上传云端
  ↓
存储 / 索引 / 清洗
  ↓
场景挖掘
  ↓
标注 / 自动标注 / 质检
  ↓
训练集 / 评测集 / 回归集
  ↓
模型训练
  ↓
离线评测 / 仿真评测 / 实车评测
  ↓
模型部署
  ↓
线上问题继续回流
```

## 3. 车端数据采集

车端会持续产生：

```text
Camera 图像 / 视频
LiDAR 点云
Radar 目标
GNSS / IMU 位姿
CAN / 车辆信号
感知输出
预测输出
规划轨迹
控制指令
HMI / 用户事件
接管事件
系统日志
```

采集时关注：

```text
什么时候录？
录哪些 topic / channel？
录多长时间？
是否压缩？
是否脱敏？
是否上传？
是否触发事件采集？
```

采集方式主要有两类。

### 3.1 全量采集

优点：

```text
数据完整
方便复盘
不容易漏掉上下文
```

缺点：

```text
存储成本高
上传成本高
筛选压力大
```

### 3.2 事件触发采集

只在关键事件前后采集数据。

触发事件包括：

```text
驾驶员接管
急刹
急加速
碰撞风险
规划失败
感知异常
系统报错
功能退出
用户投诉
```

典型做法：

```text
接管前 30 秒 + 接管后 10 秒
急刹前 10 秒 + 急刹后 5 秒
系统异常前后若干秒
```

这种方式更适合量产车辆大规模数据回流。


一般进入：

```text
PostgreSQL / MySQL / ClickHouse / Elasticsearch
```

## 5. 数据存储与索引

常见分层存储：

```text
对象存储：保存原始日志、图片、点云、视频
关系型数据库：保存任务、车辆、版本、权限等业务信息
ClickHouse / Doris：保存大规模时序指标和评测结果
Elasticsearch / OpenSearch：做场景、标签、事件检索
数据湖 Iceberg / Hudi / Delta Lake：管理大规模结构化数据表
```

索引的价值是让用户可以查询：

```text
上海雨夜接管场景
30km/h 以下无保护左转失败场景
某版本模型误检锥桶的样本
规划急刹且前方无障碍物的片段
和某个 badcase 相似的 1000 个场景
```

平台用户不会直接打开几 TB 日志找问题，所以数据闭环平台必须把原始数据转成可检索资产。

## 6. 数据清洗与治理

车端回来的数据常见问题：

```text
日志包损坏
topic 缺失
时间戳异常
传感器丢帧
坐标系错误
版本信息缺失
重复上传
数据不完整
标注格式不统一
隐私信息未脱敏
```

数据治理要判断：

```text
数据能不能用
数据适合做什么
数据质量等级如何
是否需要修复、丢弃、重采或人工检查
```

常见质量检查：

```text
文件完整性检查
topic 完整性检查
topic 频率检查
时间戳单调性检查
多传感器同步检查
图像是否黑屏 / 花屏
点云是否稀疏 / 缺失
定位是否漂移
CAN 信号是否异常
软件版本是否匹配
```

脏数据的危害：

```text
进入训练，会污染模型
进入评测，会误导结论
进入仿真，会导致回归结果不可信
```

## 7. 场景挖掘

场景挖掘解决的问题是：

```text
从海量数据中找出值得训练、标注、评测、复盘的关键片段。
```

### 7.1 挖掘对象

驾驶行为类：

```text
急刹
急加速
急转向
频繁变道
低速蠕行
规划犹豫
靠边停车
```

风险事件类：

```text
接管
碰撞风险
TTC 过低
AEB 触发
误刹
漏刹
功能退出
```

道路结构类：

```text
路口
环岛
匝道
隧道
施工区
收费站
无车道线道路
异形路口
```

交通参与者类：

```text
行人横穿
非机动车逆行
大车遮挡
加塞车辆
临停车辆
锥桶
施工人员
```

感知 / 规划 badcase：

```text
目标漏检
目标误检
车道线错误
红绿灯识别错误
预测轨迹错误
规划过保守
规划激进
控制抖动
```

### 7.2 挖掘方法

```text
规则引擎
时序数据分析
空间检索
多模态检索
Embedding 检索
异常检测
数据去重
```

三层能力：

```text
规则挖掘：基于明确条件筛选场景
统计 / 时序挖掘：发现系统异常和指标异常
Embedding / 多模态检索：通过相似性召回长尾样本
```

## 9. 数据集构建与版本管理

常见数据集：

```text
训练集：用于训练模型
验证集：用于调参和模型选择
测试集：用于离线评估
回归集：用于每次版本升级做稳定性检查
黄金集：高质量、长期固定的核心评测集
长尾集：覆盖 rare case / corner case
```

数据集必须版本管理，因为模型结果需要可追溯：

```text
这个模型用了哪些数据？
数据标注版本是什么？
训练配置是什么？
代码版本是什么？
训练环境是什么？
评测指标是什么？
```

核心认知：

```text
模型版本 = 代码版本 + 数据集版本 + 配置版本 + 训练环境 + 指标结果
```

## 12. 仿真与回归

典型链路：

```text
真实路测数据
  ↓
场景抽取
  ↓
场景泛化
  ↓
OpenSCENARIO / OpenDRIVE
  ↓
仿真回放
  ↓
指标评测
  ↓
问题回流
```

仿真解决：

```text
复现 badcase
批量测试模型
做参数扰动
生成长尾场景
做版本回归
降低实车测试成本
```

回归集用于检查：

```text
旧问题有没有修复？
旧能力有没有退化？
新模型是否引入新风险？
```

没有回归集，模型迭代就会变成“修一个问题，引入三个问题”。

## 16. 模块化智驾与 E2E 智驾下的数据闭环差异

### 16.1 模块化智驾的数据闭环

更偏向：

```text
按模块发现问题
按模块构建数据
按模块训练评测
按模块定位 badcase
```

例如：

```text
感知漏检 → 补感知数据
预测错误 → 补交互轨迹
规划失败 → 补规划场景
控制抖动 → 分析控制指标
```

### 16.2 E2E 智驾的数据闭环

更偏向：

```text
按场景发现问题
按整体驾驶表现评估
按相似场景召回数据
按闭环仿真验证效果
```

例如：

```text
模型在雨夜左转失败
  ↓
召回类似雨夜左转场景
  ↓
加入训练集和回归集
  ↓
训练 E2E 模型
  ↓
闭环仿真验证
```

E2E 越强，数据闭环的重要性越高，因为模型能力主要来自数据分布、场景覆盖和训练评测体系。

## 18. 数据闭环难点

### 18.1 数据太多，但有价值数据太少

大部分路测数据是普通场景。

真正有价值的是：

```text
异常
长尾
接管
失败
高风险
模型不确定
用户不满意
```

所以筛选能力比单纯存储能力更重要。

### 18.2 标注成本高

尤其是：

```text
3D 框
多帧轨迹
Occupancy
BEV 标签
复杂场景语义
```

需要自动标注和主动学习降低成本。

### 18.3 评测不等于真实体验

离线指标变好，不代表实车一定变好。

例如：

```text
mAP 提升，但规划体验没提升
接管率下降，但误刹增加
仿真通过，但真实路口失败
```

所以需要：

```text
离线评测 + 仿真评测 + 封闭场测试 + 实车灰度
```

### 18.4 数据、模型、版本关系复杂

如果没有严谨版本管理，会出现：

```text
不知道哪个数据训练了哪个模型
不知道哪个模型跑了哪个评测
不知道线上问题对应哪个代码版本
```

### 18.5 长尾场景永远挖不完

道路世界是开放环境。

新城市、新天气、新道路、新施工、新交通行为都会产生新问题。

所以数据闭环不是一次性项目，而是长期基础设施。

## 19. 一个完整案例：接管数据闭环

假设某车在路口发生接管，完整闭环如下：

```text
1. 车端检测到接管事件
2. 自动保存接管前 30 秒和后 10 秒日志
3. 上传 rosbag / record / MCAP 到云端
4. 元数据入库：城市、道路、天气、版本、车型、时间
5. 解析 topic：感知、定位、规划、控制、CAN、HMI
6. 数据质量检查：topic 是否完整，时间戳是否异常
7. 场景挖掘：识别为无保护左转 + 对向车交互
8. 自动召回相似场景
9. 送入标注平台补充目标、车道、行为标签
10. 构建无保护左转 badcase 数据集
11. 训练感知 / 预测 / 规划或 E2E 模型
12. 离线评测新模型
13. 将该接管片段转为仿真回归场景
14. 新模型在仿真中重跑
15. 若通过评测，进入灰度发布
16. 上线后继续监控类似路口接管率
```

这说明：

```text
一次失败，不只是一次问题；
它会变成数据、标签、训练样本、评测用例、仿真场景和版本回归资产。
```

---

## 20. 最终总结

智能驾驶数据闭环可以理解为三层能力。

### 20.1 数据资产化

```text
把车端原始日志变成可存储、可检索、可理解的数据资产。
```

### 20.2 问题样本化

```text
把接管、误刹、漏检、规划失败等问题变成可训练、可评测、可复现的样本。
```

### 20.3 系统进化

```text
通过训练、评测、仿真、部署、回流，让智驾系统持续变强。
```

最终可以用一句话概括：

```text
数据闭环不是简单收集数据，而是建立“发现问题 → 构造数据 → 修复模型 → 验证效果 → 再次回流”的自动化进化系统。
```
